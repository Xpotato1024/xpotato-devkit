# xpotato-devkit 改善提案書

## 1. 背景
`xpotato-devkit` は、AI支援開発におけるトークン消費削減と安全性向上を目的とした、repo-agnostic な CLI ツールキットである。現行 README でも、差分集計、エンコーディング検査、ブロック抽出・置換、コミット/PR テンプレ生成などを通じて、LLM 使用時のトークン消費量を抑える設計思想が示されている。 fileciteturn19file0

今回のアップグレードでは、従来の課題であった block 機能の弱さや Markdown 更新支援の不足を大きく補強しており、ツール全体は「便利な補助 CLI」から「AI に渡す文脈を圧縮する基盤」へ一段進んだと評価できる。

---

## 2. 現状評価
今回の更新内容は、方向性としてかなり良い。特に次の点が強い。

### 2.1 block 系の強化
- 構造解析により、関数・型・見出しなどの局所抽出精度が向上
- `block outline` により本文を渡さず API 構造だけ把握可能
- `block context` により patch 作成に必要な最小文脈のみ取得可能

これらは、AI にファイル全体を読ませる必要を減らし、探索段階のトークン消費削減に直結する。

### 2.2 Markdown 更新の専用化
- `md append-section / replace-section / ensure-section / append-bullet`
- frontmatter 保持
- 重複除去対応

これにより、実装記録や benchmark 記録の一部更新を CLI で安全に処理できる。  
特に Gale のように「毎実装で Markdown 記録も更新する」開発フローと高相性である。

### 2.3 記録テンプレ自動生成
- `doc impl-note / benchmark-note`
- `diff summarize` との連携

これにより、実装記録の初稿を deterministic に寄せやすくなり、AI には文章整形や補足説明だけを任せる運用が可能になる。

### 2.4 patch-first / AI-agent 運用の強化
- `patch diagnose`
- `--brief`

この方向は実戦的であり、長文コード再出力より patch を優先する運用に合う。  
また、AI エージェントが結果だけを把握する際、`--brief` は累積トークン削減に効く。

---

## 3. 残る論点
一方で、改善余地もまだある。特に重要なのは次の 3 点である。

### 3.1 効果測定がまだ弱い
現状の最大の課題は、**本当に高速化・トークン削減に効いているかを定量で追えていないこと**である。

機能追加が進むほど、
- Python 起動コスト
- 文字列処理コスト
- ファイル I/O
- git subprocess 呼び出しコスト

が蓄積しやすくなる。  
そのため、トークンは減っても CLI 自体が重くなっていれば、開発体験全体では損失が出る可能性がある。

### 3.2 `--brief` は強いが、失敗時情報の設計が重要
成功時に 1 行で済ませるのは有効だが、失敗時に情報を削りすぎると再試行回数が増え、結果的にトークンも開発時間も増える。  
したがって「成功時は極小」「失敗時は compact だが再生成に必要な要点は残す」という設計が望ましい。

### 3.3 機能増加に対して UX の一貫性が重要
機能が増えるほど、
- どのコマンドを使うべきか
- どの順で使うべきか
- 出力形式が揃っているか

が重要になる。  
とくに `outline` → `context` → `extract` → `replace` の流れが一貫していれば強いが、オプション名や出力形式が揺れると、人間にも AI にも扱いにくくなる。

---

## 4. 提案
上記を踏まえ、次フェーズでは「便利機能追加」よりも「観測と一貫性」に重点を置くべきである。

### 提案 1: 実行時間計測オプションの導入
最優先提案。

#### 内容
- 主要コマンドに `--time` を追加
- 標準出力ではなく stderr に出力
- 将来的に `--time-json` も追加
- 必要に応じて簡易内訳も追加
  - total_ms
  - git_ms
  - io_ms
  - render_ms

#### 理由
何度も実行する CLI は、1 回あたりが小さくても累積コストが大きい。  
また、Python 実装が本当に足を引っ張っているか、あるいは git や I/O が支配的かを事実ベースで切り分ける必要がある。

#### 期待効果
- Rust への再実装優先度を感覚ではなく計測で判断できる
- どのコマンドが遅いかを特定できる
- 改善の ROI を定量化できる

---

### 提案 2: `--brief` の失敗時ポリシー明確化
#### 内容
- 成功時: `OK: ...`
- 失敗時: `FAIL: ...` に加え、再生成や再実行に必要な最小限の要因を含める
- patch diagnose などは hunk 単位の compact 理由を統一フォーマットで返す

#### 理由
AI エージェントは短い出力を好むが、情報を削りすぎると再試行で余計なコストが発生する。

#### 期待効果
- 短さと再現性の両立
- エージェント連携時の失敗ハンドリング改善

---

### 提案 3: コマンド体系と出力形式の統一
#### 内容
- `outline / context / extract / replace` の UX を統一
- 可能なら行番号表示・symbol 指定・brief 出力の設計をそろえる
- Markdown 系も `append / replace / ensure` で命名・返却形式を揃える

#### 理由
ツールの実力は高まっても、使い方がぶれると AI に覚えさせにくくなる。

#### 期待効果
- AGENTS.md や Skills に書きやすくなる
- AI のコマンド選択ミスが減る
- 人間の学習コストが下がる

---

### 提案 4: 利用実績の軽量観測
#### 内容
- 任意 opt-in で、コマンド使用回数や平均時間をローカル保存
- 例: `.devkit-metrics.jsonl`
- 収集対象は匿名・ローカル限定
  - command
  - duration_ms
  - brief_mode
  - success/fail

#### 理由
「どの機能が本当に使われているか」が分からないと、改善優先度を誤る。

#### 期待効果
- 実使用ベースのロードマップ策定
- 不要な機能肥大化の抑制
- 高頻度コマンドへの最適化集中

---

### 提案 5: トークン削減効果の定性・定量検証
#### 内容
- 代表的ワークフローを定義
  - repo 構造把握
  - 対象関数の探索
  - patch 用 context 抽出
  - 実装記録更新
- devkit なし / ありで、入力文字量または token estimate を比較
- README または docs に実測例を記載

#### 理由
このツールの価値は「便利」ではなく「AI 協業時の圧縮効率」にある。  
そこを示せると、思想と成果が一致する。

#### 期待効果
- README の説得力向上
- ユーザーにとっての導入判断がしやすくなる
- どのコマンドが真に価値を生んでいるか見える

---

## 5. 優先順位
次の順で進めるのが妥当である。

1. `--time` / `--time-json` 導入
2. `--brief` の失敗時設計整理
3. コマンド UX / 出力形式の統一
4. ローカル利用実績の軽量観測
5. トークン削減効果の比較検証と文書化

---

## 6. 最終結論
今回の xpotato-devkit のアップグレードはかなり筋が良く、  
特に

- block 系の構造化
- outline / context による探索圧縮
- Markdown 更新の専用化
- 実装記録テンプレ生成
- patch-first 運用の強化
- `--brief` によるエージェント向け省トークン化

は高く評価できる。

そのうえで、次段階で最も重要なのは、さらに機能を積むことではなく、  
**「その機能が本当に速く、どれだけトークン削減に効いているかを測ること」**である。

したがって、次の改善軸は

**観測の導入**  
**失敗時を含めた短出力設計の洗練**  
**UX の一貫性向上**

の 3 点に置くのが最も合理的である。
