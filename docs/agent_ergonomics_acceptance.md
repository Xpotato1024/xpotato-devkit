# Agent Ergonomics 受け入れ基準
## 実際の混在ワークツリーを持つリポジトリにおける devkit-first ワークフロー

更新日: 2026-04-04

---

## 目的

この文書は、リポジトリ作業中にエージェントが最初に選ぶ決定的なツールとして `devkit` を使えるようにするための受け入れ基準を整理したものです。

ここでの要件は、混在ワークツリーを持つ実運用の Rust リポジトリでの利用経験に基づいています。ただし、目指す挙動は汎用的である必要があり、Gale 固有の名前、パス、成果物フォーマットに依存してはいけません。

---

## 目標

`devkit` は、人間とコーディングエージェントの双方にとって、次の流れを自然に実行できるようにするべきです。

1. 長い生のパッチ出力を読まずに、関連する差分を要約する
2. 長いファイルから必要なブロックだけを確認する
3. 意図した差分範囲に基づいてコミット文や PR 文面を下書きする
4. 非対話環境でも安全にブランチを push する

---

## 受け入れ基準

### 1. `devkit diff summarize`

#### 現在の課題
- 未ステージ変更の概要把握には有用
- ただし、ステージ済みだけを対象にしたい場合や、比較範囲を明示したい場合には弱い

#### 受け入れ条件
- `devkit diff summarize --staged` でステージ済み変更のみを要約できる
- `devkit diff summarize --base <ref> --head <ref>` で明示した比較範囲を要約できる
- 出力はコンパクトで、LLM が扱いやすいプレーンテキストのままである
- 差分が 0 件のケースが明示的に分かる

### 2. `devkit block extract`

#### 現在の課題
- 対象の見出しや関数名がすでに分かっている場合には有効
- ただし、分かっているのが「そのファイルが長い」ということだけなら、追加の探索手順がまだ必要になる

#### 受け入れ条件
- `devkit block extract --list-headings <file>` で Markdown 見出しを一覧できる
- `devkit block extract --list-functions <file>` でベストエフォートの関数名一覧を取得できる
- 見出しまたは関数の検索に失敗した場合、近い候補が提案される
- 出力は決定的で、LLM のコンテキストに貼り付けやすい

### 3. `devkit git commit-message`

#### 現在の課題
- 下書き生成自体は有用
- ただし、混在ワークツリーではステージ済みのみ、または明示した範囲だけを元に下書きしたいことが多い

#### 受け入れ条件
- `--staged` でステージ済み変更のみから下書きできる
- `--base/--head` または `--commits` で差分元を明示的に絞り込める
- 差分が空のケースでは、誤解を招くテンプレートを出さず明確に失敗する
- 日本語と英語のテンプレートが簡潔かつ安定している

### 4. `devkit git pr-body`

#### 現在の課題
- テンプレート生成自体は有用
- ただし、意図したブランチやコミット範囲だけを対象に PR 本文を生成するのが必要以上に難しい

#### 受け入れ条件
- `--staged`、`--base/--head`、または `--commits` で差分元を絞り込める
- 生成されたテンプレートに、使用した差分元を短く示す注記が含まれる
- 出力は Markdown のみで、`gh pr create --body-file` にそのまま渡せる

### 5. `devkit git safe-push`

#### 現在の課題
- 安全性の挙動自体は良い
- ただし、対話確認は非対話のエージェントフローでは速度低下要因になる

#### 受け入れ条件
- `--yes` または `--no-confirm` により非対話実行ができる
- upstream 未設定時に `--remote <name>` で設定できる
- 確認を無効にしていても、保護されたデフォルトブランチへの push は引き続き拒否される
- 成功時の出力で、対象リモートと追跡設定の結果が明確に分かる

---

## ドキュメント整合性

上記の挙動が実現されれば、リポジトリ固有のエージェント向け指示として次の案内が現実的になります。

- 長い差分を読む前に `devkit diff summarize` を使う
- 長いファイル全体を開く前に `devkit block extract` を使う
- 日本語の下書きには `devkit git commit-message` と `devkit git pr-body` を使う
- ブランチの push には `devkit git safe-push --remote origin --yes` を使う

CLI がこれらのパターンをきれいに支えられない場合、リポジトリ側の指示やローカルスキルは弱めに留め、フォールバック動作を明示的に記述するべきです。

---

## 推奨する実装順

1. `diff summarize` にスコープ付き差分入力を追加する
2. 同じ差分スコープ指定を `git commit-message` と `git pr-body` に追加する
3. `block extract` に探索補助を追加する
4. `safe-push` の非対話挙動が文書化され、安定していることを確認する

---

## 完了条件

この提案は、エージェントが次のコマンドを使って混在ワークツリーの変更を処理できるようになった時点で完了です。

- `devkit diff summarize`
- `devkit block extract`
- `devkit git commit-message`
- `devkit git pr-body`
- `devkit git safe-push --remote origin --yes`

その際、リポジトリのポリシーで明示的に求められる場合を除き、生の git コマンドや場当たり的なファイル読み取りへフォールバックしないことを条件とします。
