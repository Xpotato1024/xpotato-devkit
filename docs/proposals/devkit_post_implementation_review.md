# devkit Post-Implementation Review

## 概要

本ドキュメントは、`xpotato-devkit` の Rust 実装完了後に、公開向け仕上げの観点から現状を再確認したレビューです。  
結論として、**Rust 化そのものは完了しており、README・CI・legacy 整理も大きく前進している**一方で、まだ **ブランチ運用整合** と **Releases 実運用明確化** に重要な課題が残っています。

---

## 良くなった点

### 1. README の主語が Rust に切り替わった

README では、`xpotato-devkit` の主実装が **Rust 製ネイティブ CLI** であることが明示されています。  
また、Python 版は後方互換のために残しているだけであり、新機能追加の対象は Rust 版のみであることも明記されています。

### 2. Python legacy の位置づけが整理された

Python 版は `legacy/python/` に退避され、README 側でも legacy 扱いであることが整理されています。  
加えて、`docs/legacy/python-legacy.md` が用意され、以下の方針が明文化されています。

- 新機能追加は行わない
- 致命的なバグ修正のみ対象
- 可能な限り Rust 版の利用を推奨
- 将来的な削除可能性もある

これは二重保守の抑制という意味で非常に良い整理です。

### 3. CI が Rust 主体に整理された

`.github/workflows/ci.yml` では、Rust を前提とした以下のジョブが整備されています。

- 3 OS（Ubuntu / Windows / macOS）での test
- `cargo fmt -- --check`
- `cargo clippy -- -D warnings`
- release build による smoke test

この状態であれば、少なくとも「Rust 実装は置いてあるだけ」ではなく、継続的に検証される土台ができています。

### 4. root 表層の Python 主体感はかなり薄れた

以前は root に Python パッケージとして見える資産が残っており、公開画面からは「Rust 移行済み」に見えにくい状態でした。  
今回の確認では、少なくとも表層上はその問題がかなり改善されています。

---

## まだ残る重要指摘

## 1. 既定ブランチと README / CI の整合が取れていない

現状のリポジトリ既定ブランチは **`master`** です。  
一方で、README のインストール例は `--branch main` になっており、CI も `push` / `pull_request` ともに `main` を監視対象にしています。

このズレは小さく見えて、実運用上はかなり問題です。

### 問題点

- README 通りに `cargo install --git ... --branch main` を実行すると失敗する可能性がある
- 実際の開発が `master` で行われている場合、CI が想定どおり走らない可能性がある
- バッジやワークフローが存在しても、既定ブランチとの整合が取れていないと信頼性を損なう

### 対応案

#### 案 A: 既定ブランチを `main` に統一する
最も自然です。README、CI、将来のテンプレート運用とも整合しやすくなります。

#### 案 B: `master` を維持する
その場合は、README・CI・各種スクリプト・将来の release workflow まで含めて `master` に揃える必要があります。

### 推奨

**`main` への統一を推奨**します。  
理由は、今後の公開リポジトリ運用・外部貢献・テンプレート流用・各種自動化との相性が良いためです。

---

## 2. Releases 導線はあるが、配布運用の実態が不明瞭

README には GitHub Releases からバイナリをダウンロードする導線があります。  
しかし、今回確認できた範囲では **release workflow の存在までは確認できませんでした**。

つまり現状は次のどちらかです。

- すでに手動で release を作っている
- まだ README だけ先行しており、自動配布は未整備

どちらでも致命傷ではありませんが、公開物としては曖昧さが残ります。

### 問題点

- 「バイナリ配布しています」と README に書いてあっても、実際の更新フローが不明だと信頼性が落ちる
- タグ運用・署名・各 OS アセット生成・命名規則が曖昧だと、利用者視点で使いづらい
- 将来的に配布更新が属人的になる

### 対応案

#### 案 A: release workflow を追加する
タグ push または GitHub Release 作成をトリガーに、

- Windows / Linux / macOS バイナリをビルド
- アセットを添付
- 必要なら checksum も生成

という形にする。

#### 案 B: しばらく手動運用を明記する
まだそこまで整えない場合は、README に「Releases は当面手動更新」と明記した方が誠実です。

### 推奨

中長期的には **自動 release workflow の追加** を推奨します。  
ただし、今すぐそこまでやらない場合でも、README の記述は実運用と一致させるべきです。

---

## 総合評価

現状の `xpotato-devkit` は、少なくとも以下の点で「Rust 移行済みの CLI」として十分に成立しています。

- 主実装が Rust であることを README で明示
- Python 版を legacy として整理
- Rust 前提の CI を整備
- 公開向けの見え方が大きく改善

したがって、前回の「中身は進んでいるが表層が移行途中に見える」という段階からは、明確に前進しています。

一方で、残る課題はもはや大規模実装ではなく、**公開運用の整合性** に集中しています。  
このため、次の優先順位は非常に明確です。

---

## 優先順位付きアクション

### 優先度 High

1. **既定ブランチ名を実運用に合わせて統一する**
2. **README の install 例を実運用と一致させる**
3. **CI の対象ブランチを実運用と一致させる**

### 優先度 Medium

4. **Releases の配布フローを明文化する**
5. **可能であれば release workflow を追加する**

### 優先度 Low

6. **将来的に checksum / signing / install script などを検討する**
7. **ベンチマーク結果を README か docs に追加し、Rust 化の価値を定量化する**

---

## 結論

`xpotato-devkit` は、**Rust 実装完了後の公開仕上げとしてかなり良い位置まで来ている**。  
ただし、最後の詰めとしては「新機能追加」よりも、**ブランチ運用・CI・README・Releases の整合性を揃えること**の方が重要です。

ここを揃えれば、単なる「Rust 化済み」ではなく、**公開可能な Rust 製 CLI ツールチェーン**としてかなり説得力のある状態になります。
