# xpotato-devkit 改善ロードマップ案
xpotato-devkit はすでに、差分要約・ブロック抽出/置換・テンプレ生成を通じて、AI支援開発におけるトークン消費抑制を狙う設計になっている。README でもその目的が明示されている。

一方で、現状の block 機能はヒューリスティック寄りであり、関数終端も空行ベースで判定しているため、複雑なコードや長い文書では局所編集の精度に限界がある。

そのため、今後の改善は「AIに渡す文脈を意味単位で最小化する」ことを主軸に置く。

---

## 目的
- Codex / LLM への入力文脈を最小化し、トークン消費を抑える
- 長いコードや Markdown 全体ではなく、必要な局所だけを安全に扱えるようにする
- Gale のような「毎実装ごとにドキュメント更新も伴う開発フロー」と高相性な基盤に進化させる
- patch-first / diff-first な AI 開発運用を強化する

---

## 基本方針
改善方針は次の 2 本柱とする。

1. 局所編集の精度を上げる
2. 記録更新を定型化する

これにより、コード編集だけでなく、実装記録・ベンチ結果記録・PR/commit 文生成まで含めたトークン最適化を狙う。

---

# Phase 1: 局所編集の精度向上
最優先フェーズ。  
現状の block extract / replace は有用だが、関数抽出が正規表現ベース、終端も空行ヒューリスティックであるため、精度面に限界がある。

## 1-1. 構造ベース抽出の導入
対象:
- Python: `def`, `class`
- Rust: `fn`, `impl`, `struct`, `enum`, `mod`
- Go: `func`, `type`, `interface`

改善内容:
- 正規表現頼みではなく、字下げや波括弧を考慮した構造解析を導入する
- 可能であれば AST ベースで抽出する
- `block extract --symbol name`
- `block replace --symbol name`
  のような意味単位指定を追加する

期待効果:
- ファイル全体ではなく必要な関数/型だけを AI に渡せる
- 誤抽出や置換範囲ズレを減らせる
- 「局所編集のために全文を読む」ケースを減らせる

## 1-2. Markdown 見出し抽出の安定化
現状でも Markdown heading 指定はあるが、より実運用向けに強化する。

改善内容:
- 見出しの完全一致 / 部分一致 / パス指定を分ける
- `# / ## / ###` の階層単位で安全に section を抽出する
- 同名見出しが複数ある場合に disambiguation を出す
- `--list-headings` の結果からそのまま再利用しやすい ID 指定を導入する

期待効果:
- 実装記録や roadmap 更新の誤置換を防ぐ
- 長い Markdown を section 単位で安定運用できる

## 1-3. dry-run 出力の強化
改善内容:
- 置換前後の差分サマリを表示
- 対象範囲のプレビュー
- 「一致候補が複数ある」場合の警告
- 適用不能時の理由を詳細化

期待効果:
- AI と組み合わせた半自動更新でも事故率を下げられる
- patch 適用前の確認がしやすくなる

---

# Phase 2: ドキュメント更新の定型化
あなたのように、毎実装ごとに Gale 記録を書いている運用では、ここが非常に重要。  
トークン消費の多くはコード編集だけでなく、記録更新にも発生するため、このフェーズの費用対効果は高い。

## 2-1. Markdown 更新専用コマンド
新設候補:
- `devkit md append-section`
- `devkit md replace-section`
- `devkit md ensure-section`
- `devkit md append-bullet`

改善内容:
- 指定見出しの末尾に追記
- 見出しがなければ作成
- frontmatter を壊さず更新
- 箇条書きの重複除去や整形補助

期待効果:
- 「記録文書の一部だけ追記したい」用途を CLI で閉じられる
- AI に全文を書き換えさせる必要が減る

## 2-2. 実装記録テンプレート生成
現状は commit-message / pr-body テンプレがあるが、実装記録テンプレもあると相性が良い。

新設候補:
- `devkit doc impl-note`
- `devkit doc benchmark-note`
- `devkit doc review-note`

生成内容例:
- 変更概要
- 変更ファイル
- 背景
- 実装内容
- 検証
- 残課題

期待効果:
- 毎回似た構成の記録を書く運用に強い
- AI に渡す指示が短くて済む
- 文体・章立ての揺れを抑えられる

## 2-3. 差分からの記録下書き生成
改善内容:
- `diff summarize` の結果を元に Markdown 下書きを生成
- changed files / additions / deletions を記録テンプレへ自動注入
- 必要に応じて git log / diff stat も埋め込む

期待効果:
- 実装記録の初稿作成を deterministic に寄せられる
- AI には「文章を整える」部分だけを頼める

---

# Phase 3: AI向け compact prompt / patch-first 強化
xpotato-devkit の思想をさらに明確化するフェーズ。  
ポイントは「AI に完成コード全文を書かせる」のではなく、「差分と最小文脈だけを渡す」こと。

## 3-1. compact prompt 生成
新設候補:
- `devkit prompt edit-target`
- `devkit prompt summarize-change`
- `devkit prompt doc-update`

出力項目例:
- 対象ファイル
- 編集対象ブロック
- 変更目的
- 守るべき制約
- 出力形式
- 参照してよい関連ファイル

期待効果:
- 毎回人間が長い指示文を書く必要がなくなる
- Codex に渡すプロンプトの冗長性を削減できる

## 3-2. patch apply の強化
現状の patch は `git apply` の薄いラッパーであり、方向性は良いが情報量が少ない。

改善内容:
- 失敗 hunk の詳細表示
- 部分適用
- `--reject` 相当の補助
- patch 適用可能性の診断レポート
- 再生成のための短いエラー要約出力

期待効果:
- AI に「全文再出力」ではなく「patch の修正」だけを返させやすくなる
- patch-first workflow を実運用に乗せやすい

## 3-3. 依存最小コンテキスト収集
将来的な高難度改善として導入を検討する。

改善内容:
- 編集対象シンボルに対して
  - 定義
  - 呼び出し元
  - 型定義
  - 関連テスト
  - import / use
  を最小限で束ねる

期待効果:
- 必要十分な文脈だけ AI に渡せる
- 大規模 repo でもトークン効率を上げやすい

# 優先順位まとめ
## 最優先
1. 構造ベース / AST ベースの block extract / replace
2. Markdown section 更新専用機能

## 次点
3. 実装記録テンプレ生成
4. compact prompt 生成

## その次
5. patch apply 強化
6. 依存最小コンテキスト収集

---

# 最終結論
xpotato-devkit の今後の改善は、「便利機能を増やす」方向よりも、
**AIに渡す文脈を意味単位で最小化する基盤**として磨くのが最も良い。

特に Gale のような
- 毎実装でコード変更がある
- 毎実装で記録 Markdown も更新する
- diff / benchmark / review artifact を扱う
という運用に対しては、

**局所編集の高精度化**  
+  
**記録更新の定型化**

この 2 本柱が最も費用対効果の高い改善になる。
